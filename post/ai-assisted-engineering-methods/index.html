2026年4月20日：AI 辅助工程方式

2026-04-20 • 2 分钟阅读 •

读了15位工程师分享各自做法后，沉淀出我自己认可并想践行的方式。

我对 AI 辅助工程的基本判断

AI 不减少迭代，反而增加迭代。 每次迭代更快，所以能负担更多轮次。接受第一个输出就交付，是最大的陷阱。

核心技能转移：从"写"到"审"。 看出细微 bug、不必要的抽象、AI 自信但错误的判断——这需要经验，无法被替代。

设计文档是最重要的产物。 AI 从它生成代码，文档质量直接决定输出质量。投在设计上的时间永远值得。

AI 是力量倍增器，不是判断的替代品。 我仍然是那个决定"这段代码该不该存在"的人。

我的工作流

需求 / 规划

用 AI 反复质问自己的想法：挑战假设、挖掘细节、找出非目标。不是让 AI 替我想，而是让它把我没想到的漏洞逼出来。先写清楚问题陈述、目标用户、成功指标，再让 AI 来攻击这些假设。

设计（投入最多的阶段）

编码之前，让 AI 给两个可行方案，自己权衡决策。简单的项目也要做多轮设计迭代——AI 能快速构建代码，但修不了烂架构。

大功能用结构化 spec 文件记录：目标、方法变更、类型变更、测试变更。这一步防止 AI 实现时偏轨，也给自己一个在代码写出前发现设计问题的机会。

编码前先在 plan mode 里来回讨论，确认方案再动手。

实现

几乎把所有编码委托给 AI，自己专注在设计和判断上。大任务拆开，多个 agent 并行跑不同部分。

循环是：生成 → 运行 → 观察 → 修复。不做"大爆炸式"提交，每一步可控。

测试

不用写代码的同一个 agent 来测试。另起一个 agent，专门找 edge case 和 negative case。

关键路径偏向集成测试，不堆大量脆弱的单元测试。

代码审查

多 agent 从不同角度审：安全性、性能、正确性、简洁性。让 AI 自审：“像高级工程师一样审查这个”——这一步能持续捕捉生成时漏掉的问题。

分几轮走：正确性/可维护性 → 安全性 → 设计/性能风险。

合并永远是我自己做的决定。

上下文管理

把工作标准写进 CLAUDE.md：命名规范、提交流程、PR 模板、代码质量检查。不在每次 session 里重复解释。

每次 session 开始加载项目架构文档，让 AI 有正确的心智模型再动手。

我认可的工具分工

用途 工具
主力交互编码 Claude Code (CLI)
PR 自动审查 Claude Code / Copilot
设计参考 Figma MCP
CI 集成 GitHub Actions + Claude Code

工具不是越多越好。关键是每个工具有明确的角色，不要用同一个 agent 又写代码又测试。

几句话值得贴在桌子上

AI might be better at writing code, but I’m better at knowing whether the code should exist at all.

The biggest productivity gain isn’t faster coding; it’s spending more time on design, because implementation is no longer the bottleneck.

The difference is discipline, not tooling.
